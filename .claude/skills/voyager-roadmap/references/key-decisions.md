# Agreed Technical Decisions — Voyager Buddy

This file is the authoritative record of every design decision that was explicitly discussed and agreed during roadmap planning. Do not deviate from these without flagging it as a conscious change.

---

## Phase 1 — Architectural Foundations

### Dual-layer architecture
- **Decision: Approach B** — add a parallel reactive layer on top of the existing eval-based execution. Do NOT rewrite Node.js to a goal-based API in Phase 1.
- Approach A (goal-based) is the long-term migration target but is too large a change for Phase 1. Skills continue to be eval'd as JS. Reactive layer runs alongside.

### Interruption model
- **Decision: Adaptive (priority-based)** — not purely transactional, not purely soft.
- Five tiers:

| Priority | Name | Trigger | Type | Polling |
|---|---|---|---|---|
| 0 | CRITICAL | Fire, lava, void | Immediate event-driven | — |
| 1 | HIGH | Health ≤ 3 hearts, drowning | Soft (abort flag, next yield) | 200 ms |
| 2 | MEDIUM | Hunger ≤ 4, hostile ≤ 6 blocks | Transactional (finish current action) | 500 ms |
| 3 | LOW | Hunger ≤ 8, hostile ≤ 15 blocks | Inform only | 2000 ms |
| 4 | STRATEGIC | All other decisions | Normal LLM cycle | — |

- Priority 0 uses Mineflayer event listeners (`bot.on('onFire')`), not polling.
- Soft interrupt: abort flag checked inside skill code loops. Skill throws `AbortError` on detection.

### Python ↔ Node.js communication for reactive events
- **Decision: Variant C** — Node.js reacts autonomously. Python is NOT notified in real time.
- Python discovers reactive events from the `recentReactiveEvents` array in the next `/step` response.
- Do NOT add WebSocket, do NOT add a `/status` polling endpoint.

### Player observation
- **Decision: Variant A** — use only what Mineflayer can natively observe (position, health, equipped items).
- Do NOT implement a companion Fabric mod in Phase 1 or Phase 2.
- Player inventory is not observable — this is intentional and consistent with real Minecraft mechanics.

### Survival override layer
- **Decision: Variant B** — Python pre-filter in `voyager.py` before `CurriculumAgent.propose_next_task()`.
- Deterministic check: health < 6 → override, hunger < 6 → override, hostile + no weapon → override.
- This is not an LLM call. It is a hard rule that prevents the LLM from ignoring critical survival state.

### Reset modes
- **Decision:** three modes: `"none"` | `"soft"` | `"hard"`.
- Survival sessions default to `"none"` — no inventory clear, no kill.
- `"hard"` retained for testing only.

## Phase 2 — Survival Core

### Shelter building approach
- **Decision: Approach A** — LLM generates shelter code. No hardcoded `buildSimpleShelter` primitive.
- Rationale: the bot should learn to build through experience, just as it learned to mine. A hardcoded primitive removes the learning opportunity.
- The `pillarUp` primitive is the emergency fallback for early nights while the LLM skill is still developing.
- Building restriction removed from `curriculum.txt` for survival mode.

### `isSheltered` detection
- Block directly above bot is solid (not air, not leaves) AND at least 3 of 4 horizontal adjacent blocks are solid AND at least 1 light source within 5 blocks.

### Fight/flee decision logic
- **Decision: multi-factor**:
  - health < 8 → flee unconditionally
  - mob is creeper AND distance < 5 → flee (explosion risk)
  - no weapon equipped → flee + try to equip
  - spider/cave_spider AND count > 1 → flee
  - weapon equipped AND health ≥ 8 → engage (killMob primitive)
  - else → flee

### Flee mechanic
- **Decision: human-like escalation** — sprint away from mob cluster (10-15 blocks), then `pillarUp(4-5 blocks)` if still pursued.
- If home is within 50 blocks: sprint toward home instead.
- Do NOT flee in a random direction.
- Spider cap (place 1 block on top of pillar) is learned through experience, not hardcoded.

### `pillarUp` primitive
- New control primitive in `voyager/control_primitives/pillarUp.js`.
- Parameters: `(bot, material, height = 4)`.
- Selects material from inventory (cobblestone > dirt > any block).

### Home concept
- **Decision: Variant B** — explicit position, stored in `ckpt/survival/home.json`.
- Set by bot after shelter construction (`set_by: "bot"`).
- Overridable by player via chat: `"set home here"` → `set_by: "player"`, takes permanent precedence.

### Episodic survival memory
- **Decision:** new checkpoint file `ckpt/survival/experiences.json` + Python module `voyager/agents/survival_memory.py`.
- Events: death, significant damage, successful shelter.
- Lesson field generated by skill model (gpt-5.4-nano) — short LLM call, low cost.
- Last N lessons injected into CurriculumAgent prompt as "Recent survival experiences".
- This is what drives experience-based building decisions (e.g., "spider climbed pillar → need cap").
- **Implemented details:**
  - Death attribution via `bot.lastDamagingEntity` cache (updated on `entityHurt`, 3-second window) — NOT nearest mob at death time (skeletons attack from 15 blocks).
  - Significant damage threshold: > 4 HP (2 hearts) per hit. Captures zombie (2h), skeleton (2-3h), creeper blast. Filters chip damage.
  - Shelter lesson gated on `isSheltered=True` AND no hostile mobs within 6 blocks at recording time.
  - `_isBeingReset` flag on bot suppresses death recording during hard reset `/kill @s`.
  - Deduplication: skip `record_event` if last entry has identical type+cause.
  - `block.emittedLight > 0` scan for light check (NOT `block.light` which is emitted-only and 0 for stone).
  - All door types counted as solid walls regardless of open/closed state (simplification).

### Night override vs pillarUp
- **Decision:** Python override for shelter fires at `noon`/`sunset` only — NOT at night.
- At night, `pillarUp` from the reactive layer (Phase 2.2) is the fallback. Interrupting nighttime with LLM-driven construction causes infinite retry loops and failed builds.
- Shelter building is a DAYTIME task (prepare before dark). Night survival = reactive layer.

### Session resume
- **Decision:** `RESUME` env var in `.env` controls `resume=True/False` in Voyager constructor.
- NEVER run with `RESUME=false` if `ckpt/*/vectordb/` directories already exist — assertion error (vectordb count ≠ JSON count).
- To reset task progress: delete curriculum JSON files AND vectordb together. Never one without the other.

### Food priority
- Saturation-first: cooked beef/porkchop (12.8) > cooked mutton/chicken (9.6) > bread (6.0) > cooked fish (6.0) > apple (2.4) > raw meat (1.8).
- Always prefer cooked over raw within the same tier.

### Chat commands (Phase 2 scope)
- Rule-based parser only in Phase 2. LLM parser replaces it in Phase 4.
- Parser interface: `parseCommand(message)` → `{action, args}` — designed so Phase 4 only swaps the function, not the pipeline.
- Supported: `come`, `stop`, `follow me`, `stay`, `go home`, `set home here`, `what are you doing`, `inventory`, `give me [item]`, `craft [item]`.
